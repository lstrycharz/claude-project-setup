# Multi-Asset Project — Validation Checklist

Run against the full asset set before declaring the project done — not against individual assets in isolation. The aim is to catch consistency issues that only surface when assets are reviewed together.

## Voice and tone consistency

- [ ] All assets share the same voice (per `voice-and-standards.md`)
- [ ] Tone matches the project's audience across every asset
- [ ] No forbidden words present in any asset
- [ ] Voice exceptions (per asset type) are documented and intentional

## Messaging consistency

- [ ] Key takeaway phrased the same way across assets
- [ ] Calls-to-action use consistent language
- [ ] Anchor metric or stat is the same number, formatted the same way, in every asset
- [ ] Audience is named consistently (no asset says "users" while another says "customers")

## Cross-references

- [ ] Cross-asset links work and point to the right places
- [ ] Asset titles/subject lines match between paired assets (e.g., email subject = landing page H1)
- [ ] Each asset references the central project hub if required

## Coordination

- [ ] All required asset types in the project are present
- [ ] Each asset includes its required elements (per `voice-and-standards.md`)
- [ ] Versioning is clean — no asset references a stale version of a stat or phrase

## Polish

- [ ] No placeholders or `[TBD]` markers remain in any asset
- [ ] All assets pass their individual quality bars (typo-free, well-formatted)
- [ ] Final approval/handoff steps complete per coordination rules

---

*Adjust as your team's standards evolve. Each item is a yes/no question — vague items don't belong in a checklist.*
